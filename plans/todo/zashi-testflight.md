# Full Voting Round Lifecycle in Zashi

## Context

Zashi currently only handles two states: no active round (show "Create Test Round") and ACTIVE round (delegation → voting → shares). Rounds transition ACTIVE → TALLYING → FINALIZED automatically on-chain, but the app can't see those states — `/rounds/active` returns 404 once a round leaves ACTIVE. For TestFlight testing with a deployed chain, we need visibility into all phases.

**Round management is handled externally.** The `shielded_vote_generator_ui` (React web app, hosted on Vercel) handles creating voting rounds, managing proposals, and publishing to the chain. Zashi is a voter client only — it discovers rounds, delegates, votes, and views results. No round creation UI in Zashi.

**Service discovery via CDN.** Vote servers (chain RPC + helper — single merged binary) and nullifier IMT providers are listed in a JSON config file hosted on a CDN. Zashi fetches this config at startup to know which servers to talk to. This avoids hardcoding URLs and makes it easy to add/remove nodes without app updates.

**Session creation is gated by VoteManager.** PR #92 added the VoteManager role on-chain — `MsgCreateVotingSession` requires `sender == VoteManager`. This is set via `MsgSetVoteManager` (any bonded validator can bootstrap the first one). The `shielded_vote_generator_ui` submits sessions as standard Cosmos SDK transactions signed by the VoteManager address. No SDK-side changes are needed — the on-chain role check is sufficient.

### Ceremony prerequisite

A **key exchange ceremony** was added to main. `CreateVotingSession` now requires `ceremony.Status == CONFIRMED` — without it, the chain rejects session creation with `ErrCeremonyNotReady`.

For a single-validator local setup, the ceremony completes automatically:

1. **Register Pallas key** — `MsgRegisterPallasKey` (standard Cosmos SDK tx; validator's Pallas public key, generated by `init.sh` via `pallas-keygen`)
2. **Deal EA key** — `MsgDealExecutiveAuthorityKey` (standard Cosmos SDK tx; dealer ECIES-encrypts `ea_sk` to each validator's Pallas pk, publishes `ea_pk`)
3. **Auto-ack** — `PrepareProposal` automatically decrypts the ECIES payload and injects `MsgAckExecutiveAuthorityKey`. With one validator, a single block confirms the ceremony.

After step 3, the chain is in `CONFIRMED` state and round creation works. The EA public key (`ea_pk`) is pulled from ceremony state — Zashi never needs to know it.

**For local testing**, run `make ceremony` after the chain is up. This is a standalone Rust test that calls the existing `bootstrap_ceremony` function (idempotent — skips if already CONFIRMED):

```bash
# Terminal 1: init and start the chain
cd sdk && make init && make start

# Terminal 2: bootstrap ceremony (one-time, ~10s)
cd sdk && make ceremony

# Now the chain is ready for round creation from the generator UI or test-e2e.
```

**Ceremony status** is queryable at `GET /zally/v1/ceremony`.

### Available SDK query endpoints

The SDK already serves all the round and tally query endpoints we need:

| Endpoint | Returns | Notes |
| --- | --- | --- |
| `GET /zally/v1/rounds` | All rounds (any status) | New — not yet used by iOS |
| `GET /zally/v1/rounds/active` | First ACTIVE round or 500 | Currently the only one iOS uses |
| `GET /zally/v1/round/{round_id}` | Single round by hex ID | New — not yet used by iOS |
| `GET /zally/v1/tally-results/{round_id}` | Finalized decrypted tallies | New — iOS only has per-proposal `fetchProposalTally` |
| `GET /zally/v1/ceremony` | Ceremony state + ea_pk | Exists |
| `GET /zally/v1/vote-manager` | Vote manager address | Exists |

The `VoteRound` proto now includes `Proposals []*Proposal` where each proposal has `Options []*VoteOption` (with `Index` and `Label`). The iOS `Proposal` model doesn't carry options yet — needs updating.

## Changes

### 0. Service discovery via CDN config

Two service types need to be discovered at runtime:

| Service                  | Current value               | Location                          | Notes                                   |
| ------------------------ | --------------------------- | --------------------------------- | --------------------------------------- |
| Vote servers (RPC+helper)| `http://localhost:1318`     | `VotingAPIClientLiveKey.swift`    | Merged binary serves both chain API and helper endpoints |
| Nullifier IMT providers  | `http://46.101.255.48:3000` | `VotingStore.swift:22`            | Separate service for nullifier exclusion proofs          |

Since the SDK and helper server are now a single merged binary, one URL serves all chain endpoints (rounds, delegation, cast-vote, commitment tree, tally) and all helper endpoints (share submission, ZKP #3 generation). No separate helper server list needed.

**Hosting:** Static JSON file in `shielded_vote_generator_ui/public/voting-config.json`. This is auto-deployed to Vercel on merge to main, served at `https://zally-phi.vercel.app/voting-config.json`. The Vercel URL is the only hardcoded URL in the Zashi iOS binary — can be updated via an app release, but the server list it points to can change anytime.

**Config file (`shielded_vote_generator_ui/public/voting-config.json`):**

```json
{
  "version": 1,
  "vote_servers": [
    { "url": "http://46.101.255.48:1318", "label": "Primary" }
  ],
  "nullifier_providers": [
    { "url": "http://46.101.255.48:3000", "label": "Primary" }
  ]
}
```

`vote_servers` entries each serve the full set of endpoints:
- Chain API: `/zally/v1/rounds`, `/zally/v1/rounds/active`, `/zally/v1/round/{id}`, `/zally/v1/ceremony`, `/zally/v1/delegate-vote`, `/zally/v1/cast-vote`, `/zally/v1/commitment-tree/*`, `/zally/v1/tally-results/*`
- Helper API: `/api/v1/shares` (receives share payloads, generates ZKP #3, submits reveal-share TXs)

`nullifier_providers` entries serve: `/nullifier/proof/{nullifier_hex}` (IMT non-membership proofs for verifying notes were unspent at snapshot height)

**`VotingServiceConfig.swift`** — New model:

```swift

    public let version: Int
    public let voteServers: [ServiceEndpoint]
    public let nullifierProviders: [ServiceEndpoint]

    public struct ServiceEndpoint: Codable, Equatable, Sendable {
        public let url: String
        public let label: String
    }
}
```

**`VotingAPIClientInterface.swift`** — Add:

```swift

```

**`VotingAPIClientLiveKey.swift`** — Implement: fetch from `https://zally-phi.vercel.app/voting-config.json` (hardcoded constant), decode, return. Cache in memory for the session. All subsequent API calls use URLs from this config.

**`VotingStore.swift`** — On `.initialize`:

1. Fetch service config from CDN
2. Store the full list of vote servers and nullifier providers
3. Pick a primary vote server for RPC calls (first reachable), pick first nullifier provider
4. Use these URLs for all subsequent API calls
5. Share submission uses the full `vote_servers` list (see section 8a)

For local development, fall back to hardcoded localhost defaults if the CDN fetch fails.

### 4. Rework initialization to handle all statuses

**`VotingStore.swift`** — The `.initialize` action changes:

1. Open voting DB (same as today)
2. Fetch service config from CDN → `.serviceConfigLoaded`
3. Configure API client URLs from resolved config
4. Fetch all rounds via `fetchAllRounds()`
5. Pick the most relevant round:
   - If any ACTIVE round exists → use it
   - Else if any TALLYING round exists → use it
   - Else if any FINALIZED rounds exist → use the most recent one
   - Else → no round
6. If round found, send `.activeSessionLoaded(session)`
7. `.activeSessionLoaded` routes based on `session.status`:
   - `.active` → proceed with notes/hotkey loading as today, start status polling
   - `.tallying` → set `screenStack = [.tallying]`, start status polling
   - `.finalized` → set `screenStack = [.results]`, fetch tally results
8. If no round at all → `activeSession` stays nil, show no-round UI

For ACTIVE rounds, after notes are loaded (`.votingWeightLoaded`):
- If `notes.isEmpty` → navigate to `.ineligible` (reason: no notes at snapshot)
- If `totalWeight < 12_500_000` → navigate to `.ineligible` (reason: balance too low)
- Otherwise → proceed with witness verification and delegation as today

The key refactor: split the `.initialize` effect so notes/hotkey loading only happens for ACTIVE rounds. For TALLYING/FINALIZED, we skip straight to the appropriate screen.

### 5. Round status polling

**`VotingStore.swift`** — `.startRoundStatusPolling`:

### 8. Distribute shares across vote servers

Currently all 4 encrypted shares are POSTed to a single hardcoded helper server in a sequential loop (`VotingAPIClientLiveKey.swift ~line 281`). The design doc (Phase 4-5) envisions share delegation to servers for **temporal unlinkability** — servers stagger reveal-share submissions so an observer can't correlate 4 shares to the same voter by timing.

**New behavior:** Distribute the 4 shares across available vote servers from the CDN config. Each vote server runs the same merged binary and can independently generate ZKP #3 and submit the reveal-share TX to the chain.

**Distribution logic (in `VotingAPIClientLiveKey.swift` or `VotingStore.swift`):**

```
let servers = serviceConfig.voteServers
if servers.count >= 4:
    // One share per server (first 4 servers, shuffled)
    assign share[i] → servers.shuffled()[i]
elif servers.count > 1:
    // Round-robin across available servers
    assign share[i] → servers[i % servers.count]
else:
    // Single server — all 4 shares go there (current behavior)
    assign all shares → servers[0]
```

Each share payload is self-contained (contains `all_enc_shares` for the shares hash, plus the specific `enc_share` being revealed), so any server can process any share independently.

**Privacy benefit:** With N servers, no single server sees all 4 shares from the same voter. Combined with each server's independent timing decorrelation, this significantly reduces the ability to link shares back to a single voter.
